The 30-Day Practice Plan

Follow this schedule in your sandbox environment to build muscle memory.

Week 1

Foundations (Days 1-7)

  • Day 1: SU01, SU01D - Create a dialog user, view user details (simulate approvals).
  • Day 2: SU10, PFUD - Perform a mass lock and run PFUD on a sample user.

Core Security T-Codes & Playbooks

User Management: SU01, SU10, SU01D

SU01: Create/change/lock/unlock user. Real Example: New AP clerk needs Z_FI_AP_CLERK role.

SU10: Mass maintenance. Example: Lock 150 contractors after expiry. (Always test first).

SU01D: Display user read-only. Perfect for providing audit evidence of role validity.

Troubleshooting: SU53, SU56, PFUD

SU53: Displays the last failed authorization check. Critical: Must be run immediately after the error.

SU56: Shows authorizations loaded into the user buffer (in-memory). Use if role is assigned but user still lacks access.

PFUD: User Master Comparison. Recreates profiles/syncs user master after role changes or transports.

Playbook for Mass Issue after Transport: Check transport log, apply PFUD, run profile generator in PFCG, check AGR_1251 entries comparing DEV vs PRD.

Role & Config Maintenance: PFCG, SU24, RZ10/11

PFCG: Core role maintenance. Manage menu, authorizations (S_TCODE, org restrictions), and generate profiles.

SU24: Controls proposed authorizations for a T-code. If a custom T-code (Z_INVOICE) needs new auth objects, update SU24 so PFCG pulls them automatically.

RZ10/RZ11: Profile and runtime params. RZ11 is dynamic (runtime), RZ10 is persistent and requires a restart (e.g., enforcing password complexity).

SAP GRC Access Control Mastery

GRC Access Control is typically used via the NWBC web UI, supported by backend GRAC* transactions.

Access Risk Analysis (ARA) & Mitigation

Concept: Detects SoD violations. If a user has "Create Vendor + Pay Vendor", ARA flags it.

Remediation Steps:

  1. Check for business justification. If none, remove offending role via PFCG.
  2. If justified and cannot be split, create a Mitigation Control in GRC.
  3. Assign an owner and periodic review frequency to the control.
  4. Re-run ARA and capture the report as evidence.

Access Request Management (ARM)

Workflow: Login to NWBC -> Open ARM -> Fill requester, role, reason -> Submit -> Check MSMP approval path.

Troubleshooting Provisioning Failures: Check ARM request -> provisioning log -> RFC connectivity -> connector user privileges -> re-process or manually provision temporarily.

Emergency Access Management (EAM / Firefighter)

T-Code: GRAC_EAM

Workflow: Provides centralized emergency IDs. A Controller must review the session, download the event log, highlight risky T-codes executed, and sign off or escalate to security.
